### 1.2 Flujo de Datos (Request → Response)

```
Cliente (Browser)
    │
    │  1. HTTP Request (JSON + Bearer Token header)
    ▼
[FastAPI Router]
    │
    │  2. Middleware CORS verifica origen
    │  3. Dependencia get_current_user() valida JWT
    │     - Decodifica token → extrae user_id
    │     - Si expirado: 401 Unauthorized
    ▼
[Router Handler] (ej: POST /gastos)
    │
    │  4. Pydantic schema valida el request body
    │     - Si inválido: 422 Unprocessable Entity
    ▼
[SQLAlchemy Session]
    │
    │  5. ORM construye query SQL
    │  6. SQLite3 ejecuta la transacción
    │  7. Retorna ORM model instance
    ▼
[Router Handler]
    │
    │  8. Pydantic response_model serializa la respuesta
    ▼
Cliente (Browser)
    │  JSON response (200 / 201 / 204 / 4xx / 5xx)
```

### 1.3 Decisiones Técnicas Justificadas

#### SQLite3 vs PostgreSQL

| Criterio | SQLite3 | PostgreSQL |
|---|---|---|
| Setup | Archivo local, cero configuración | Servidor separado, requiere init |
| Concurrencia | Un escritor a la vez (suficiente: 1 usuario) | Multi-usuario, concurrencia alta |
| Backup | `cp gastos.db backup.db` | pg_dump, configuración separada |
| Recursos Docker | ~5MB RAM | ~50-100MB RAM mínimo |
| Portabilidad | El DB viaja con el volumen Docker | Requiere exportar/importar |

**Decisión:** SQLite3. La app es de uso personal (1 usuario o familia). La limitación de escritura concurrente no aplica. Reduce la complejidad de infraestructura significativamente.

#### FastAPI vs Flask/Django

- **Flask:** Minimalista pero requiere librerías adicionales para validación, async, docs automáticas.
- **Django:** Demasiado opinionado, ORM propio, overhead innecesario para una API REST simple.
- **FastAPI:** Tipado nativo con Pydantic, documentación OpenAPI automática en `/docs`, async por defecto, inyección de dependencias integrada. Ideal para APIs REST modernas con Python 3.12.

#### JWT Stateless vs Sessions

- Sessions requieren almacenamiento server-side (Redis u otro). Con SQLite3 como única dependencia, agregar Redis rompe el principio de simplicidad.
- JWT stateless: el token contiene toda la información necesaria. El refresh token se almacena en DB para permitir revocación.
- Compromiso: access token corto (15min) + refresh token largo (7 días) almacenado con hash en DB.

#### Docker Compose para desarrollo local

- Garantiza paridad entre entornos (dev/prod).
- El volumen nombrado para SQLite persiste datos entre reinicios del contenedor.
- Un solo comando (`docker-compose up`) levanta frontend + backend.

### 2.1 Schema DDL Completo

```sql
PRAGMA journal_mode = WAL;
PRAGMA foreign_keys = ON;
PRAGMA encoding = 'UTF-8';

-- =============================================
-- TABLA: users
-- =============================================
CREATE TABLE IF NOT EXISTS users (
    id          INTEGER PRIMARY KEY AUTOINCREMENT,
    email       TEXT    NOT NULL UNIQUE,
    password_hash TEXT  NOT NULL,
    full_name   TEXT,
    created_at  TEXT    NOT NULL DEFAULT (datetime('now')),
    updated_at  TEXT    NOT NULL DEFAULT (datetime('now'))
);

CREATE INDEX IF NOT EXISTS idx_users_email ON users(email);

CREATE TRIGGER IF NOT EXISTS trg_users_updated_at
AFTER UPDATE ON users
BEGIN
    UPDATE users SET updated_at = datetime('now') WHERE id = NEW.id;
END;

-- =============================================
-- TABLA: categorias
-- user_id NULL = categoría global del sistema
-- user_id NOT NULL = categoría custom del usuario
-- =============================================
CREATE TABLE IF NOT EXISTS categorias (
    id          INTEGER PRIMARY KEY AUTOINCREMENT,
    user_id     INTEGER REFERENCES users(id) ON DELETE CASCADE,
    name        TEXT    NOT NULL,
    color       TEXT    NOT NULL DEFAULT '#6B7280',
    icon        TEXT    NOT NULL DEFAULT 'tag',
    created_at  TEXT    NOT NULL DEFAULT (datetime('now')),
    UNIQUE(user_id, name)
);

CREATE INDEX IF NOT EXISTS idx_categorias_user ON categorias(user_id);

-- =============================================
-- TABLA: gastos
-- =============================================
CREATE TABLE IF NOT EXISTS gastos (
    id           INTEGER PRIMARY KEY AUTOINCREMENT,
    user_id      INTEGER NOT NULL REFERENCES users(id) ON DELETE CASCADE,
    amount       REAL    NOT NULL CHECK(amount > 0),
    category_id  INTEGER NOT NULL REFERENCES categorias(id) ON DELETE RESTRICT,
    description  TEXT,
    fecha        TEXT    NOT NULL DEFAULT (date('now')),
    created_at   TEXT    NOT NULL DEFAULT (datetime('now'))
);

CREATE INDEX IF NOT EXISTS idx_gastos_user_fecha ON gastos(user_id, fecha);
CREATE INDEX IF NOT EXISTS idx_gastos_user_category ON gastos(user_id, category_id);
-- Índice compuesto para la query del dashboard (filtro por mes)
CREATE INDEX IF NOT EXISTS idx_gastos_user_fecha_month
    ON gastos(user_id, strftime('%Y-%m', fecha));

-- =============================================
-- TABLA: refresh_tokens
-- Almacena hash del refresh token para revocación
-- =============================================
CREATE TABLE IF NOT EXISTS refresh_tokens (
    id          INTEGER PRIMARY KEY AUTOINCREMENT,
    user_id     INTEGER NOT NULL REFERENCES users(id) ON DELETE CASCADE,
    token_hash  TEXT    NOT NULL UNIQUE,
    expires_at  TEXT    NOT NULL,
    revoked     INTEGER NOT NULL DEFAULT 0 CHECK(revoked IN (0,1)),
    created_at  TEXT    NOT NULL DEFAULT (datetime('now'))
);

CREATE INDEX IF NOT EXISTS idx_refresh_tokens_user ON refresh_tokens(user_id);
CREATE INDEX IF NOT EXISTS idx_refresh_tokens_hash  ON refresh_tokens(token_hash);
```

## 3. Sistema de Autenticación JWT

### 3.1 Estructura de Tokens

**Access Token (TTL: 15 minutos)**
```json
{
  "sub": "42",
  "email": "usuario@ejemplo.com",
  "type": "access",
  "iat": 1712700000,
  "exp": 1712700900
}
```

- `sub`: user_id como string (estándar JWT)
- `email`: evita lookup adicional a DB en cada request
- `type`: distingue access de refresh en validación

**Refresh Token (TTL: 7 días)**
```json
{
  "sub": "42",
  "type": "refresh",
  "jti": "uuid-v4-unico",
  "iat": 1712700000,
  "exp": 1713304800
}
```

- `jti` (JWT ID): UUID único por token, se almacena su hash en `refresh_tokens`. Permite revocación individual.

### 3.2 Flujo Completo del Ciclo de Vida

```
[Login]
   │  POST /auth/login {email, password}
   │  1. Busca user por email
   │  2. bcrypt.verify(password, hash)
   │  3. Genera access_token (15min) + refresh_token (7d)
   │  4. Guarda hash(refresh_token) en refresh_tokens table
   │  5. Retorna ambos tokens
   ▼
[Uso normal]
   │  Cada request: Authorization: Bearer <access_token>
   │  get_current_user():
   │    - jwt.decode(token, SECRET_KEY)
   │    - Si exp < now: raise 401
   │    - Retorna user_id del payload
   ▼
[Refresh — cuando access_token expira]
   │  POST /auth/refresh {refresh_token: "..."}
   │  1. Decodifica refresh_token
   │  2. Verifica type == "refresh"
   │  3. Calcula hash(jti) → busca en refresh_tokens
   │  4. Verifica revoked == 0 y expires_at > now
   │  5. Marca token anterior como revoked = 1
   │  6. Genera nuevo par access_token + refresh_token
   │  7. Guarda nuevo hash en refresh_tokens
   │  8. Retorna nuevos tokens
   ▼
[Logout — v1: client-side]
   │  El cliente descarta ambos tokens del storage
   │  (La revocación server-side es post-v1)
```

### 3.3 Dependencia FastAPI: `get_current_user`

```python
# Contrato de la dependencia (no implementación)
# Se inyecta con: current_user: User = Depends(get_current_user)
#
# Flujo:
# 1. Extrae header: Authorization: Bearer <token>
# 2. jwt.decode(token, settings.SECRET_KEY, algorithms=["HS256"])
# 3. Valida type == "access"
# 4. Retorna instancia User del ORM
# 5. Si cualquier paso falla: raise HTTPException(401)
```

### 3.4 Hashing de Contraseñas

- **Algoritmo:** bcrypt con `rounds=12`
- **Librería:** `passlib[bcrypt]`
- **Por qué bcrypt sobre argon2:** bcrypt está battle-tested desde 1999, amplio soporte en todas las plataformas, y `rounds=12` es el balance óptimo entre seguridad (>100ms por hash en hardware moderno) y performance para una app personal. Argon2 es superior en teoría pero la diferencia práctica es irrelevante a esta escala.

**Auth requerida:** No

**Request body:**
| Campo | Tipo | Requerido | Descripción |
|---|---|---|---|
| email | string | Sí | Email único del usuario |
| password | string | Sí | Mínimo 8 caracteres |
| full_name | string | No | Nombre para mostrar |

**Response 201:**
```json
{
  "access_token": "eyJ...",
  "refresh_token": "eyJ...",
  "token_type": "bearer",
```

| Código | Descripción |
|---|---|
| 422 | Validación fallida (email inválido, password corto) |

## 5. Docker

### 5.1 `docker-compose.yml`

```yaml
version: '3.9'

services:
  backend:
    build:
      context: ./backend
      dockerfile: Dockerfile
    ports:
      - "8000:8000"
    volumes:
      - sqlite_data:/data
      - ./backend:/app  # hot-reload en desarrollo — remover en prod
    environment:
      - DATABASE_URL=sqlite:////data/gastos.db
      - SECRET_KEY=${SECRET_KEY}
      - ACCESS_TOKEN_EXPIRE_MINUTES=${ACCESS_TOKEN_EXPIRE_MINUTES:-15}
      - REFRESH_TOKEN_EXPIRE_DAYS=${REFRESH_TOKEN_EXPIRE_DAYS:-7}
      - CORS_ORIGINS=${CORS_ORIGINS:-http://localhost:5173}
    healthcheck:
      test: ["CMD", "curl", "-f", "http://localhost:8000/health"]
      interval: 30s
      timeout: 10s
      retries: 3
    restart: unless-stopped

  frontend:
    build:
      context: ./frontend
      dockerfile: Dockerfile
    ports:
      - "5173:5173"
    environment:
      - VITE_API_URL=http://localhost:8000/api/v1
      - VITE_DATA_SOURCE=${VITE_DATA_SOURCE:-api}
    depends_on:
      backend:
        condition: service_healthy
    restart: unless-stopped

volumes:
  sqlite_data:
    name: control_gastos_sqlite
```
